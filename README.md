# Nginx Log Security Monitoring Dashboard

## 概要

本プロジェクトはAWS EC2上にNginx環境を構築し、Webアクセスログを解析・可視化するダッシュボードを開発したものです。

当初は単純なログ解析およびアクセス集計ツールとして設計していましたが、実際のログを分析する中で、特定IPからの大量404エラーやスキャン系リクエストの偏りを確認しました。

この観測をきっかけに、単なるアクセス解析ではなく「異常アクセス検知」の観点が必要だと判断し、攻撃パターン分類およびリスクスコアリング機能を追加しました。

結果として、本プロジェクトはWebアクセス解析に加え、簡易的なセキュリティ監視ダッシュボードへと発展しました。

---

## 目的

- Nginxアクセスログの構造を理解する  
- アクセス傾向とエラー発生状況を可視化する  
- 異常アクセス（スキャン・攻撃）の兆候を検知する  
- ログ解析を通じてサーバー運用の基礎を習得する  
- Pythonによるデータ処理と可視化の実践力を身につける  

---

## 環境

- クラウド: AWS EC2 (Ubuntu Server 22.04)  
  - 仮想サーバーを構築し、Nginxを用いてWebサーバーを公開  
  - セキュリティグループによりHTTP/SSH通信を許可
- OS: Ubuntu Server 22.04  
- Webサーバー: Nginx  
- 開発言語: Python 3.x  
- 使用ライブラリ: matplotlib, collections（defaultdict）  

---

## 実装内容

### 1. アクセスログ解析
- IPアドレスごとのアクセス数を集計  
- HTTPステータスコード別の件数を集計  

---

### 2. 攻撃パターン分類
アクセスURLをもとに以下を分類：

- phpunit系攻撃スキャン  
- Laravel関連スキャン  
- WordPress探索（wp-）  
- admin / login など管理画面探索  
- CGI・xmlrpcなどの脆弱性探索  
- その他（unknown）

---

### 3. リスクスコアリング
IPごとに以下のルールでスコア化：

- アクセスごと +1  
- 404エラー +2  
- スキャン系URL +5  
- phpunitアクセス +10  

→ 危険度の高いIPをランキング表示  

---

### 4. 可視化機能
- matplotlibによるIPアクセス棒グラフ生成  
- HTMLダッシュボード出力  
- 分析結果の可視化

## 苦労したこと（課題と改善）

- ログフォーマットの違いにより、IP・URL・ステータスコードの抽出位置が安定せず解析結果がずれる問題が発生した  
  → Nginxログ構造に基づき抽出ロジックを固定化  

  確認コマンド：
  tail -f /var/log/nginx/access.log

---

- Tabとスペース混在によりPythonのIndentationErrorが発生  
  → インデント統一とバックアップ運用へ変更  

  対応コマンド：
  cp analyzer.py analyzer_backup.py
  expand -t 4 analyzer.py > tmp.py && mv tmp.py analyzer.py
  nano analyzer.py
  python3 analyzer.py

  ※実装当時は解析をメインに行っていたため analyzer.py という名称でしたが、現在は機能拡張により security_log_dashboard.py にリネームし、役割を「ログ解析ツール」から「セキュリティ監視ダッシュボード」へ明確化しています。

---

- 404エラーの増加が攻撃か正常か判断できず、分析設計に迷った  
  → URLパターン分類を追加（phpunit / wp- / admin / login）

---

- スコア設計が単純加算で一部IPの値が極端に大きくなる問題  
  → 重み付け方式に変更し、相対評価へ改善  

---

- unknown分類が増加し分析精度が低下  
  → scan条件を拡張し、cgi-bin / xmlrpc / env なども分類対象に追加  

---

- SSH上で直接編集したことで構文エラーや管理ミスが発生  
  → ローカルバックアップ＋段階的修正フローへ変更  

---

- サーバー上での開発により、ログ解析・Webサーバー・Python実行環境が同一となり、エラー影響範囲が広く原因特定が困難になった  
  → 処理を「ログ取得」「解析」「可視化」に分離して設計改善  

---

- 攻撃判定ロジック追加により条件分岐が複雑化し、評価順序で結果が変わる問題が発生  
  → 優先順位を明確化し、上位攻撃パターンから評価する構造に修正  

---

- 想定外のURLパターンが多数存在し単純分類が困難だった  
  → unknown分類を維持しつつ、新規攻撃パターンを追加できる拡張設計に変更
